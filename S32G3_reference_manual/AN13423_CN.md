## 在 S32G 上使用 LLCE 的 CAN2CAN, CAN2ETH 和 ETH2CAN 功能

由：NXP Semiconductors 提供

## 1. 简介

本应用指南是《LLCE 入门指南》和《LLCE 固件用户指南》的补充文档，旨在介绍如何在 S32G 中使用 CAN2CAN、CAN2ETH 和 ETH2CAN 功能。

这三者是 LLCE 实现无主机干预 CAN 网关任务（Offloading）的核心功能。LLCE 能够在这几个通道之间执行 CAN 帧路由，即：在不同的 CAN 通道间路由（CAN2CAN）、以及在 CAN 与以太网间路由（CAN2ETH / ETH2CAN），全程无需主机核（Host Core）的干预。这些功能显著降低了路由延迟并减轻了主机负担。浏览本文档后，您将理解这些功能的原理以及如何进行配置操作。

LLCE 可根据配置好的路由表进行操作，完全零 CPU 负载：
- **CAN2CAN**：当配置好的帧 ID 进入指定 CAN 通道时，LLCE 自动将其路由至目标 CAN 通道。
- **CAN2ETH**：当配置好的帧 ID 进入指定 CAN 通道时，LLCE 自动将该 CAN 帧封装为 IEEE1722 格式的太网帧及 UDP 报文，由 PFE（包转发引擎）发往以太网。
- **ETH2CAN**：当 PFE 收到以太网帧时，LLCE 会将其解析、解包（IEEE1722 / UDP 包），然后再路由至对应的 CAN 通道。

## 2. CAN2CAN, CAN2ETH 和 ETH2CAN 特性

**CAN2CAN 特性** 会依据配置好的路由表，在剥离 CPU 负载下进行 CAN 帧路由。可用选项有：
- **多播/单播 (Multicast/Unicast)**：既可指定单一目标通道（单播），也可配置多目标通道（多播）。
- **ID 重新映射 (ID remapping)**：可对 CAN 帧 ID 重映射，支持标准 ID 与扩展 ID 间的切换。
- 支持纯 Classic CAN 与 CANFD 帧之间的相互转换配置。

图 2. CAN2CAN 特性
<!-- image -->

**CAN2ETH 特性** 用于从 CAN 帧脱机封装至符合 IEEE1722 规范的以太网帧中。封装过程如下：
- CAN 帧被打包至 IEEE1722 帧中（符合 AVTP 时间同步/非时间同步控制格式，以 ACF CAN Brief 或 Full 消息打包），或者直接打包为 UDP 报文。
- LLCE 对这些信息打包并存放至 SRAM 里的缓存器。
- 包的总长是由预设的 Buffer Size 控制的。
- Stream ID 为固定值，不可配置。

图 3. CAN2ETH 特性
<!-- image -->

**ETH2CAN 特性** 用于脱机解包符合 IEEE1722 / AVTP 规范的以太网帧。过程如下：
- 系统解析进入的 IEEE1722 帧 / UDP 包，解压数据并进行路由分发。
- 单个 AVTP 帧里所能包含的 ACF CAN 帧最大数量由配置的 HTH (Hardware Transmit Handle) 数量决定。单通道最大支持一包最多 16 帧。

**注意 (NOTE):** 为了避免与主应用层上的以太网管理相冲突，请留意：LLCE 固件在 CAN2ETH/ETH2CAN 期间使用的是 PFE_HIF3。

## 4. 在 EB Tresos 中的参数配置
本小节讲解自定义配置方案的基础重点。您可以载入样例工程配置来加以变通修改。

### 4.1. 导入示例配置
启动 EB-Tresos Studio 并且导入对应 S32G2 或 S32G3 目标下的 `Tresos_CAN2CAN_Project` 或者对应 CAN2ETH的配置文件。

图 39 - 44: 配置项目的导入与重命名说明截图 
<!-- image -->

### 4.2. 配置 CAN2CAN 所需的 Llce_Af
在 `Can2CanRoutingTable` 项内，您可以自定义以下 LLCE AF 选项参数：
1. 双击 Llce_Af 列表；
2. 切引到 Can2CanRoutingTable 界面。
3. 您可以新增或删除不同的 CAN2CAN 配置用例。（图 45 - 47 展示界面使用）
在目的地链表 `CanDestinationList` 下选择预定义好的通道 ID 作为出口。默认列表中是作为单一目标也就是单播（Unicast）。如果新增了多项地址指向就是多播（Multicast）应用。而在 General 一栏内设定其他的明细。

图 48. 显示的为全局关联：须到 `CanAdvancedFeature` 项目组里去确认引用是否激活正确。
<!-- image -->

### 4.3. 配置 CAN2ETH 所需的 Llce_Af
双击进入 `Llce_Af` -> `Can2EthRoutingTable` 后配置：
封装类型（EncapsulationType）有以下模式可供选用:
- AVTP_NTSCF_BRIEF（非时间同步 IEEE1722 简明消息）
- AVTP_NTSCF_FULL（非时间同步 IEEE1722 完整消息）
- AVTP_TSCF_BRIEF（时间同步 简明消息）
- AVTP_TSCF_FULL（时间同步 完整消息）

关于 Buffer 尺寸，如果打包数量为 N：
在 NTSCF_BRIEF 中，其长度要求 `>= 26 + (N-1)*(8+payload) -1 + 72`  且 `< 26 + N*(8+payload) -1 + 72`。
同理，针对其他三种封装体都有相应的等量公式（具体详阅原文）。
注意这里的 `can_msg_payload` 为 0 ~ 16 Quadlets（4-byte 数据块）。如果您在一个 Buffer 中存放许多短数据包要注意覆盖更新率（Over-written risk）避免总线发送瓶颈发生。

### 4.4. 配置 Can controller
在默认样例里预配置了 BCAN 0,1,14,15。想要增加别的 BCAN 控制器可以对列表进行复制扩展配置，指定好节点 ID 。（见图 50 ~ 54 的波特率（Baud rate），采样点，以及各相位的填充方式配置项。）

### 4.5. 配置 CAN Hardware object (Message buffers)
到 `CanHardwareObject` 里：
1. 指定好对应的 Mask 掩码与收发匹配机制（BASIC - Mask Enable/ FULL - Exact ID Match）。
2. 指定处理目标 CAN controller ID。
3. 指定对象是作为 TX 或者 RX 模式。此项内也可定义接受通过的逻辑区间。见图 55 及更后截图说明。

## 5. 在 S32CT 中的配置
这一章节介绍如何脱离 EB-Tresos 而使用 S32DS 原生工具体 S32 Configuration Tools (S32CT) 去调整 CAN 网关配置。

### 5.1. 搭建 S32DS 3.5, RTD, 与 LLCE 整合包环境
需事先通过 Flexera 等途径完整下载获取：
- S32 Design Studio v3.5 安装包
- 面向 S32G 线下开发扩展工具包 S32DS 3.5.1 dev package
- RTD (Real Time Drivers) 4.0 升级站点补丁
- LLCE 补丁（如 1.0.5）
然后按照系统内置的 Extension & Updates 管理窗口加入（步骤参阅原文截图 55 ~ 61）。

### 5.2. 为 S32DS 加入对应底层驱动
在 Extensions 安装好后加载 RTD 与 LLCE 相关项（包含 S32G2/G3 的组件）。配置系统信任后重启 S32DS 方可生效环境。

## 6. CAN2CAN S32CT 示例工程生成流程
在准备就绪的环境下，依次： 
1. `File -> New -> S32DS Project from Example`，在范例里选用 `Can_Llce_DS_Can2Can` 并 Finish 导入。
(图 64 - 66)
2. 转至 ConfigTools 配置视角，点按 `Update Code` 让配置生成新头文件关联。起初会出现错误符号，但一旦文件同步更新错误都会被修正完毕。 
3. 在代码窗口目录，把官方编译好的可执行 LLCE FW 核心组件(bin 文件) 复制挪动到 `LLCE_BIN_DIR` 开发板文件夹区。
- **G3** 对应于 `bin/ghs/enablement` 路径。
4. 现在可以执行 Clean 并且使用 Build Project 构建新生成好的工程项。
